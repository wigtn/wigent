# 05. 회고 — 배운 것과 남은 과제

## 잘한 것

### 1. "계약 우선 개발"이 병렬 작업을 가능하게 했다

`types.ts`를 281줄로 먼저 확정한 뒤 3명이 동시에 작업했다. 결과:
- P1(백엔드), P2(프론트), P3(훅/통합)이 **거의 동시에 커밋** (13:48, 13:49, 13:49)
- 머지 시 **충돌 0건**
- 통합에 **3분**만 소요

교훈: 급할수록 인터페이스를 먼저 합의하라. 타입 정의에 투자한 5분이 나머지 3시간의 충돌을 예방했다.

### 2. 데모 임팩트를 역산한 기획

"채팅이 랜딩페이지로 바뀐다"는 데모 시나리오를 먼저 정하고, 거기에 맞는 아키텍처를 설계했다.

- 1차 AI 심사 → Slack 다크테마 UI (시각적 완성도)
- 3차 사람 심사 → Full Page Swap (WOW 팩터)
- 코드 품질 → TypeScript strict + 컴포넌트 분리

해커톤은 제품을 만드는 게 아니라 **데모를 만드는 것**이라는 점을 명확히 인식했다.

### 3. 실용적 피봇을 두려워하지 않았다

3시간 동안 4번의 전환점을 거쳤다:
1. Slack UI 채택 (13:37)
2. 라운드 → 자유토론 전환 (14:48)
3. GPT 생성 → 템플릿 즉시 렌더링 (15:06)
4. 강제 수렴 프롬프트 (15:13)

각각의 피봇이 **문제 인식 → 해결책 → 즉시 구현** 30분 이내에 완료됐다. "처음 설계를 고수하지 않는" 유연성이 핵심이었다.

### 4. Claude Code의 효과

3명이 각각 Claude Code를 사용하여:
- P2는 Slack Chat UI 10개 컴포넌트를 **1개 커밋**으로 일괄 생성
- P3는 useDebate 441줄 + 4개 컴포넌트를 **1개 커밋**으로 생성
- 프롬프트 엔지니어링 → 코드 생성 → 테스트의 사이클이 극단적으로 빨라짐

## 기술적으로 배운 것

### SSE + AsyncGenerator는 실시간 AI 앱의 최적 패턴

```
orchestrator (yield SSEEvent) → API Route (SSE 전송) → useDebate (dispatch Action)
```

이 패턴의 장점:
- **관심사 분리**: 토론 로직은 orchestrator, 전송은 route, 상태관리는 reducer
- **타입 안전**: SSEEvent → SSEEventType → Action 변환이 모두 타입 체크
- **취소 가능**: AbortController가 모든 레이어를 관통

### 프롬프트 엔지니어링은 제품 기능이다

프롬프트 관련 커밋이 전체의 약 30%를 차지한다:
- 에이전트 페르소나 설계 (PM의 말투, 디자이너의 관점)
- 토론 단계별 지시 (초반: 자유, 후반: 수렴, 마무리: 결론 강제)
- 랜딩페이지 프롬프트 v1 → v2 → v3 반복 개선
- GPT 거부 대응 (영어 프롬프트, fallback)

AI 제품에서 프롬프트는 UI만큼 중요한 사용자 경험이다.

### GPT-4o의 한계와 대응

1. **HTML 생성 거부**: GPT-4o가 보안상의 이유로 HTML을 생성하지 않는 경우 발생
   - 대응: 영어 프롬프트로 전환 + "design prototype with placeholder content" 프레이밍
2. **품질 편차**: 같은 프롬프트에도 결과 품질이 크게 다름
   - 대응: 9개 템플릿 즉시 렌더링 + GPT 결과는 optional upgrade
3. **수렴 실패**: 에이전트들이 결론 없이 계속 새 아이디어 제시
   - 대응: 25턴 이후 강제 수렴 프롬프트 + 마지막 5턴 결론 유도
4. **레이턴시**: 30턴 토론 + 랜딩페이지 생성 = 최대 10분
   - 대응: 스트리밍으로 실시간 표시 + 템플릿 즉시 렌더링

## 아쉬운 점과 남은 과제

### 1. 사용자 토론 참여 미구현
현재 사용자는 주제를 입력하고 관전만 한다. DisabledInput 컴포넌트가 있지만 실제 입력 기능은 없다. 사용자가 토론 중간에 방향을 제시하면 더 유용한 결과를 낼 수 있다.

### 2. 에이전트 커스터마이징 미구현
에이전트의 페르소나, 수, 전문 분야를 사용자가 선택할 수 없다. PM/디자이너가 고정인 것도 주제에 따라 유연해져야 한다.

### 3. 랜딩페이지 편집 기능 부재
생성된 랜딩페이지를 수정할 수 없다. 반려 → 재토론이 유일한 개선 수단인데, 토론 재시작이 비용과 시간이 크다.

### 4. 비용 최적화
30턴 토론은 GPT-4o 호출 약 35회 이상. 프로덕션에서는 모델 티어링(중요 단계만 GPT-4o, 나머지 GPT-4o-mini)이 필요하다.

### 5. 테스트 부재
3시간 해커톤 특성상 테스트 코드가 전혀 없다. orchestrator, useDebate reducer, SSE 파서 등 핵심 로직에 대한 유닛 테스트가 필요하다.

### 6. 에러 핸들링 보강
네트워크 끊김, GPT 응답 지연, 부분 JSON 파싱 실패 등의 엣지 케이스에 대한 처리가 minimal하다.

## 결론

Wigent는 "AI 에이전트가 토론하고, 그 결과가 웹페이지가 된다"는 단순한 아이디어를 3시간 만에 동작하는 프로덕트로 만든 프로젝트다.

기술적으로 의미 있는 포인트:
1. **타입 계약 기반 병렬 개발** — 3명이 충돌 없이 동시 작업
2. **AsyncGenerator + SSE + useReducer** — 실시간 AI 앱의 클린 아키텍처
3. **실용적 피봇** — 문제 발견 → 해결까지 30분 이내
4. **이중 경로 렌더링** — AI 생성의 불확실성을 템플릿으로 보완

해커톤은 "완벽한 코드"를 만드는 시간이 아니다. 제한된 시간 안에 **가장 임팩트 있는 데모**를 만들고, **작동하는 것을 우선**하며, **필요할 때 과감하게 바꾸는** 것이 핵심이다. Wigent는 그 과정을 3시간 26커밋으로 보여준다.

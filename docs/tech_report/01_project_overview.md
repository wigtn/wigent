# 01. 프로젝트 개요 — 무엇을, 왜 만들었는가

## 배경

**Build with TRAE @Seoul Hackathon (2026-03-28)**, 코딩 시간 3.5시간(12:30~16:00), 3명이 각자 Claude Code로 병렬 개발 후 merge하는 방식으로 진행했다.

## 풀고자 한 문제

> "사업 아이디어를 구체화할 때, 혼자 생각하면 편향되고 팀 토론은 시간이 오래 걸린다."

기존 AI 도구(ChatGPT 등)는 단일 응답만 제공한다. 다양한 관점의 **충돌과 수렴** 과정이 없다. 우리는 이 빈 공간을 AI 에이전트들의 자율 토론으로 채우고, 그 결과를 단순 텍스트가 아닌 **실제 랜딩페이지**로 만들어주는 서비스를 구상했다.

## 솔루션: Wigent

사용자가 주제를 던지면:

1. **PM 에이전트(고정)** + **주제별 전문가 에이전트(자동 생성)** 가 Slack 스타일 UI에서 실시간 토론
2. 토론 중 에이전트가 **퇴장**하고 새 **전문가가 자동 합류**
3. 토론 완료 후 **아이디어 기반 랜딩페이지가 자동 생성**되어 채팅 화면 자체를 대체

```
주제 입력 → 에이전트 생성 → Slack 토론 → 에이전트 교체 → 요약 → 랜딩페이지 생성
```

## 해커톤 심사 전략

심사가 3단계로 진행된다는 점이 개발 우선순위에 직접 영향을 줬다:

| 단계 | 내용 | 우리의 대응 |
|------|------|------------|
| 1차 | AI가 URL 방문 → 프론트엔드 분석 | Slack 다크테마 UI + 모션 + 반응형 |
| 2차 | 전체 코드 제출 → 코드 품질 | TypeScript strict, 컴포넌트 분리, API 설계 |
| 3차 | 사람이 체점 → 데모 임팩트 | "채팅이 랜딩페이지로 바뀐다" Full Page Swap |

## 핵심 차별점

| vs | 차별점 |
|----|--------|
| ChatGPT | 단일 응답 vs **다관점 토론 시각화** |
| 기존 브레인스토밍 도구 | 수동 vs **AI 자율 토론 → 실제 웹페이지** |
| Multi-agent 데모 | 터미널 텍스트 vs **Slack UI + 랜딩페이지 자동 생성** |

## 팀 구성과 역할 분담

3명이 각각 Claude Code를 사용해 **완전 병렬 개발**:

| 담당 | 역할 | 주요 결과물 |
|------|------|------------|
| P1 (hwcho) | 백엔드 코어 | orchestrator, prompts, SSE API, types |
| P2 (swson) | Slack Chat UI | ChatLayout, Sidebar, ChatMessage 등 10개 컴포넌트 |
| P3 (hskim) | Input/Output + Hook | TopicInput, LandingPageView, useDebate Hook |

이 구조가 가능했던 이유는 **`types.ts`를 먼저 확정**하여 공유 계약(contract)을 만들었기 때문이다.
